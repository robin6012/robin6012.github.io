種植-毛豆

Posted on 2024-08-01 Edited on 2024-08-06

毛豆種植

播種及種植時間

播種期 ： 2-3月; 8-9月
發芽適溫 ： 20-30℃
發芽時間 ： 3-5天
生長適溫 ： 20-30℃
生長期 ： 春季; 秋季
生長環境 ： 喜溫暖氣候
播種到採收 ： 2-3個月

毛豆生產技術

毛豆為大豆莢果，發育至八分滿時，長約4.5公分、寬約1.3公分以上者的鮮豆莢，此時莢果種仁外觀皆為翠綠色故稱之為green soybean，又因毛豆仁供蔬菜用亦謂之vegetable soybean，在日本則摘除葉片後，整枝連莖桿包裝出售稱為「枝豆」。

栽培管理技術

一、播種前應注事項

(一)種子預措：毛豆種子優劣影響發芽率至鉅，故應篩選粒大且無病蟲害之種子，發芽率在80%以上，以確保毛豆發芽良好，無缺株現象。

(二)注意土壤濕度：土壤濕度對毛豆發芽有很大影響，一般而言，在田間容水量百分之85至百分之90間最適宜毛豆之發芽。

二、整地

毛豆播種分整地與不整地兩種方式進行。整地栽培在播種前將土地耕犁1～2次，再用迴轉犁碎土，並整平地面，除去殘草枯枝，然後築畦及開溝，以供播種。行株距為40公分×15公分；不整地栽培是以耕耘機附掛播種機進行播種，每隔1.2公尺開溝作畦，行株距春、秋作30公分×10公分，夏作40公分×10公分。

三、播種

毛豆播種量每分地約13至15公斤，在雨季應注意收聽氣象報告，嚴防豪雨以免豆種腐爛。播種期，春作1月下旬至3月上旬，夏作6月上旬～6月下旬，秋作8月下旬至10月上旬。

四、噴施殺草劑

毛豆播種後，應在二日內即行噴施殺草劑，每公頃噴施43%拉草乳劑或34%施得圃乳劑200-300倍稀釋液1.2-1.5公升，萌芽後每公頃可噴施35%萬帥(Onecide)1000倍稀釋液1公升抑制雜草生長。使用殺草佈於地面。可達到豆田除草之目的。

五、施肥

毛豆三要素需量

毛豆生育期短(65-85)天，故肥料宜早期施用，對毛豆生長與莢果飽滿效果較佳。春作每公頃施肥量N:P2O5:K2O＝60:40-80:60公斤換算每公頃硫酸銨300公斤；氯化鉀100公斤；過磷酸鈣220～450公斤。秋作每公頃施肥量N:P2O5:K2O＝60:40-80:90公斤。換算每公頃硫酸銨300公斤；氯化鉀150公斤；過磷酸鈣：220-450公斤。

六、灌溉與排水

就毛豆水分需要及對產量影響而言，以能保持全期濕潤為佳，次為開花及莢果充實期濕潤，再次為營養生長期濕潤。故在灌水方便之處，必需在播種前灌水一次，以利發芽，植後15-20日灌二次，30-40日灌第二次，50-60日灌第四次。灌水以田間高處灌滿水後應立即排水，高燥之地因得水不便，可分區灌水，不要全區灌注，否則先灌之處積水太久，影響毛豆生育。排水不良之地應於田畦四周或田畦中央開溝以利排水。

病蟲害防治

一、病害

毛豆主要病害，有銹病、紫斑病、露菌病等，可使毛豆減產20-50%，且全年發生應切實噴藥防治。

(一)大豆銹病：發生於葉片、葉柄及莖上，初呈圓點黃綠色水浸狀病斑，後變為黃褐色，逐漸擴大，在葉表面產生顯著隆起之夏孢子堆，引起葉片枯落。

防治法：播種後20至30天，或花蕾開始萌出時施藥一次，以後每隔7-10天施藥一次，共約施藥四次，施藥時應噴於葉片上下兩面，每公頃噴施80%鋅錳乃浦可濕性粉劑400倍稀釋液2.5-3.0公斤。

(二)大豆紫斑病：發生於莖、葉片、葉柄、莢、種子等部位，葉上初呈小圓粒狀病斑，擴大接合後，呈不規則形狀，出現紫褐至暗褐色；莖及葉柄病斑顏色與葉同，多呈紡錘形或圓形；莢部病斑初呈圓形，紫褐色，後變成黑褐色，剝開豆莢可見子實呈紫紅色，種皮縐褶嚴重者裂開，子葉外露。

防治法：播種後50天左右，如發現罹病時，應即行施藥一次，以後每隔10天施藥一次，至採收前半個月為止，施藥後，如遇雨天時，應補噴一次。每公頃噴施80%錳乃浦可濕性粉劑，400倍稀釋2-3公斤。

(三)大豆露菌病：發生於葉，葉片出現4-5公厘大小淡黃色病斑，部分變褐色，背面有灰白色徵，嚴重時影響光合作用之功能。

防治法：與大豆紫斑病同。

二、蟲害

毛豆生育全期均有蟲害發生，對產量及品質影響很大，其生育初、中、後期的害蟲種類不同，防治方法亦不同。

播種時：為預防潛蠅、蚜蟲、紅蜘蛛等蟲害。可在播種時每公頃施5%二硫松粒劑20公斤或是3%加保扶粒劑40公斤。

生長期：如發生夜盜蟲及切根蟲時，每公頃可噴90%納乃得可濕性粉劑，稀釋2,000倍以防除之。

開花前：大部分豆株在將開花時(夏作35天，春作40-45天)，為防治擬尺蠖蛾，浮塵子及毒蛾幼蟲為害，每公頃可用35%安殺番乳劑稀釋1,000倍，1.5-2公升或45%一品松乳劑稀釋1,000倍，2公升或是90%納乃得可濕性粉劑，稀釋2,000倍0.5公升。噴時務必噴在豆莢部。

發生紅葉璊時，每隔7-10天施藥一次，必須自側面噴於葉之上下面，葉柄等各部位。每公頃用50%加力可乳劑稀釋1,000倍1公升。

收穫

毛豆在子實(種粒)達7-8分熟(秋作70天，春夏作75-80天)即經行機械收穫或僱工青割，搬運至集貨場，分由工人摘取子實飽滿且二粒種仁以上之豆莢，經機械篩選，人工精選皆為兩粒種仁之豆莢，長4.5公分，寬1.3公分，始達出口規格，如豆農代表的集貨場無法容納青刈的毛豆植株時，則由工人直接在田間摘取合乎出口規格毛豆果莢。毛豆亦可剝取豆仁，供為蔬菜用。剩餘之莖、葉及單粒豆莢可供作青飼料或當作綠肥。

留種

留種必須選種，選種之目的在於維持毛豆種子純度，以使其能保持高產量，優良品質及其他良好農藝性狀之遺傳物性，使栽種之品種不致因異品種，及雜草之混雜，病蟲害傳播而失去原品種之優良性狀，因而留種除注意純度外，萌芽力也要注意。

使用說明

MACRO for Tera Term
MACRO (TTPMACRO.EXE) is an interpreter of the macro language “Tera Term Language (TTL)”, which controls Tera Term and provides various functions like auto dialing, auto login and so on.

如何運行巨集文件

可執行檔 TTPMACRO.EXE 應放置在 TTERMPRO.EXE 所在的目錄中。
有兩種方法可以運行巨集檔案。

  1. 來自 Tera 術語。
    若要啟動巨集，請選擇[控制]巨集指令，然後在開啟巨集對話方塊中選擇巨集檔案。

2）來自巨集。
巨集檔案可以指定為 TTPMACRO.EXE 命令列（快速連結）的參數。例如，如果要執行巨集檔案“DIALUP.TTL”，請指定命令列（快速連結），如下所示：

TTP巨集撥號.TTL
您可以省略檔案副檔名“.TTL”。如果省略檔案名，則會出現「開啟巨集」對話方塊。為您經常使用的巨集檔案安裝圖示（捷徑）很方便。

如果您選擇方法 2)，則可以在啟動 MACRO 後，使用巨集檔案中的「connect」指令執行 Tera Term。

當巨集運作時，您可以按下「巨集」對話方塊中的對應按鈕來暫停、重新啟動和停止巨集。

注意：運行時不顯示視窗
若要運行而不顯示窗口，請使用命令列選項“/V”啟動 TTERMPRO.EXE 或 TTPMACRO.EXE。
當使用「connect」巨集指令執行 TTERMPRO.EXE 時，必須指定「/V」作為其參數。
該選項需要在不登入的情況下運行，例如使用任務。

注意：載入巨集文件
巨集檔案被一次性載入到記憶體中，並且巨集被解釋。然而，巨集命令include總是讀取指定的巨集檔案。執行網路檔案伺服器上的巨集檔案時，如果網路不通，巨集指令包含可能會失敗。

常用指令

交互命令：
clearscreen <int>：清屏。 int=0時清屏，int=1時清屏+清緩衝區

dispstr <data1> [<data2>….]：在螢幕上輸出字元。注意：data如果是數字，會取低8位元作為ASCII字元進行列印，而不是把data當成數字列印；在螢幕上輸出的字元僅本地可見，對遠端沒有影響。

fluashrecv：清空接收字元快取。防止wait指令從接收快取讀取到異常字元。

gethostname <strvar>：把主機名稱/IP位址存放到strvar字串變數裡。可以用作日誌輸出。

unlink：中斷目前巨集（腳本）和目前連線之前的連線。這裡涉及到巨集和Teraterm和單一host的關係。巨集首先連結到Teraterm，在整個TeraTerm範圍下可以發起新連線。巨集也可以連結到單一host，在單一host裡向遠端控制台發送指令。

logclose：關閉日誌文件

logopen <filename> <binary flag> <append flag> [<plain text flag> [<timestamp flag> [<hide dialog flag> [<include screen buffer flag> [<timestamp type>]：把收到的字元輸出到filename檔中，後面都是flag。

filename：不指定目錄的時候，預設輸出到teraterm的目前目錄（可用changedir變更）
binary flag：0-CR轉換為CRLF，轉義符被剔除；1-不轉換，收到什麼存什麼
append flag：0-建立新文件，覆蓋現有文件；1-建立新文件，在現有文件追加
plain text flag：1-非ASCII可見字元不輸出到log。
（一般下面不用看）

timestamp flag：1-在每一行log加入時間戳
binary flag：1-忽略plain text flag和timestamp flag
hide dialog flag：1-不顯示Log對話框
include screen buffer flag：1-目前螢幕緩衝區在log開啟後立即寫入log
timestamp type：時間戳記類型。例如0-本地時間，1-UTC時間
logpause：接下來收到的字元不要寫入log。和logstart搭配使用，logstart恢復log記錄。

logwrite <string>：寫一句string到log，string不能是空的。

send <data1> <data2>….：發送字元。 data裡面有整數時，取整數的低8位元視為ASCII字元進行傳送。

sendln <data1> <data2>….：傳送字符，最後加一個回車。

wait <string1> [<string2> …]：等待最多10個字串中任一個的出現。字串為空時代表一個任意字元。系統變數timeout/mtimeout大於零時代表最長等待時間。等待結果存放在系統變數result中，0為超時，n為第n個字串出現了。

waitln <string1> [<string2> …]：以行為單位的wait。

wait4all <string1> [<string2> …]：等待10個字串的同時出現。

waitevent <events>：等待event出現，例如events=4時代表disconnect。

waitrecv <sub-string> <len> <pos>：等待接收到符合某個條件的字串。條件具體指的是總長為len，從pos位置開始是sub-string。

waitregex <string1 with regular expression> [<string2 with regular expression> …]：等一个符合正则表达式的字符串出现。

pause <time>：像其他语言中的sleep，等待time秒。

生土司

食譜

高筋麵粉 250g (加老麵可以減少)
砂糖 15g
鮮奶 80g (因為雞蛋的大小不定,可以依狀態增加或減少)
鮮奶油 20g
雞蛋一個
奶粉 12ｇ
速發酵母 3g
鹽 3g
老麵40g (可不加)
無鹽奶油 12g

作法

夏天天氣熱，建議製作前可將材料連盆，放入冰箱冰至少半小時，鮮奶要夠冰.

  1. 將麵粉、乾酵母、砂糖與鹽倒入攪拌缸中，以慢速混合均勻
  2. 加入牛奶、鮮奶油與雞蛋混合的液體攪拌成團，先以慢速打四分鐘，再以中速打四分鐘
    這個配方的水量蠻多，製作的麵團偏黏手
    如果麵團都黏在攪拌缸中，可以先暫停，將黏在缸底的麵團刮下來再繼續攪拌
  3. 加入室溫下軟化的奶油後，低速打兩分鐘，最後以中速打三分鐘
    攪拌機打出手套膜是理想狀態，打好的麵團光滑又柔軟，而且也不黏手了
    完成的麵團終溫理想狀態為27～28度
  4. 取出麵糰滾圓後，進行第一次基本發酵60分鐘
    一小時後，用手指戳入麵團能戳出洞且不會回彈，發酵完成
  5. 取出麵團按壓排氣，分割成三份，滾圓，中間發酵15-20分鐘
    中間發酵20分鐘後，麵團又膨膨漲大了
  6. 整形後放入吐司模型，最後發酵50分鐘，上方要加上保鮮膜
    發酵後，帶蓋吐司，大概長到吐司盒八分滿就可以進爐
  7. 送入已預熱的烤箱，以上火170度、下火210度烘烤35分鐘(實測只要160度就可以了)
    每人家中設備火力不同，實際烘烤時間請以自家設備為準

介紹

本文檔給出了構成主要 Python 發行版中標準函式庫的 Python 程式碼的編碼約定。請參閱隨附的資訊 PEP，描述Python 的 C 實作中的 C 程式碼的樣式指南。

本文檔和PEP 257（文檔字串約定）改編自 Guido 的原始 Python 樣式指南文章，並添加了 Barry 的樣式指南[2]中的一些內容。

隨著時間的推移，隨著附加約定的確定以及過去的約定因語言本身的變化而變得過時，這種風格指南也在不斷發展。

許多項目都有自己的程式設計風格指南。如果發生任何衝突，則此類特定於專案的指南優先適用於該專案。

愚蠢的一致性是小頭腦的惡魔

Guido 的重要見解之一是，程式碼的讀取次數遠多於編寫次數。此處提供的指南旨在提高程式碼的可讀性並使其在各種 Python 程式碼中保持一致。正如PEP 20所說，「可讀性很重要」。

風格指南是關於一致性的。與本風格指南的一致性很重要。專案內的一致性更為重要。一個模組或功能內的一致性是最重要的。

然而，知道何時要不一致——有時風格指南的建議並不適用。如有疑問，請運用您的最佳判斷。查看其他範例並決定哪個看起來最好。請隨時詢問！

特別是：不要僅僅為了遵守此 PEP 而破壞向後相容性！

忽略特定準則的其他一些充分理由：

應用指南會降低程式碼的可讀性，即使對於習慣閱讀遵循此 PEP 的程式碼的人也是如此。
與周圍也破壞它的程式碼保持一致（可能是出於歷史原因）——儘管這也是清理別人混亂的機會（以真正的 XP 風格）。
因為相關程式碼早於指南的引入，並且沒有其他原因需要修改該程式碼。
當程式碼需要與不支援樣式指南推薦的功能的舊版 Python 保持相容時。

空格是首選的縮排方法。

製表符應僅用於與已使用製表符縮排的代碼保持一致。

最大線路長度

將所有行限制為最多 79 個字元。

對於結構限制較少（文件字串或註解）的流動長文字區塊，行長度應限制為 72 個字元。

限制所需的編輯器視窗寬度可以並排開啟多個文件，並且在使用在相鄰列中顯示兩個版本的程式碼審查工具時效果很好。

大多數工具中的預設包裝會破壞程式碼的視覺結構，使其更難以理解。選擇這些限制是為了避免在視窗寬度設定為 80 的編輯器中進行換行，即使工具在換行時在最後一列中放置了標記符號。一些基於網路的工具可能根本不提供動態換行。

命名約定

Python 函式庫的命名約定有點混亂，因此我們永遠無法使其完全一致——儘管如此，以下是目前推薦的命名標準。新的模組和套件（包括第三方框架）應按照這些標準編寫，但如果現有庫具有不同的風格，則首選內部一致性。

壓倒一切的原則

作為 API 公共部分對使用者可見的名稱應遵循反映用法而不是實現的約定。

描述性：命名風格

有很多不同的命名風格。它有助於識別正在使用的命名樣式，而與它們的用途無關。

通常區分以下命名風格：

  • b（單個小寫字母）

  • B（單一大寫字母）

  • lowercase

  • lower_case_with_underscores

  • UPPERCASE

  • UPPER_CASE_WITH_UNDERSCORES

  • CapitalizedWords（或 CapWords，或 CamelCase——如此命名是因為它的字母看起來凹凸不平[4]）。有時也稱為 StudlyCaps。

為繼承而設計

始終決定類別的方法和實例變數（統稱為“屬性”）應該是公共的還是非公共的。如有疑問，請選擇非公開；稍後將其公開比將公共屬性設為非公開更容易。

公共屬性是您期望與您的類別無關的客戶端使用的屬性，並承諾避免向後不相容的變更。非公共屬性是指那些不打算被第三方使用的屬性；您不保證非公共屬性不會更改甚至被刪除。

我們在這裡不使用術語“私有”，因為在 Python 中沒有屬性是真正私有的（通常不需要做不必要的工作）。

另一類屬性是「子類 API」的一部分（在其他語言中通常稱為「受保護」）。有些類別被設計為繼承、擴展或修改類別行為的各個方面。在設計這樣的類別時，請注意明確決定哪些屬性是公共的、哪些是子類別 API 的一部分以及哪些確實只能由基底類別使用。

考慮到這一點，以下是 Pythonic 指南：

  • 公共屬性不應有前導底線。

  • 如果您的公共屬性名稱與保留關鍵字衝突，請在您的屬性名稱後面新增一個尾隨下劃線。這比縮寫或損壞的拼字更好。 （然而，儘管有這條規則，「cls」是任何已知為類別的變數或參數的首選拼寫，尤其是類別方法的第一個參數。）

  • 對於簡單的公共資料屬性，最好隻公開屬性名稱，而不需要複雜的存取器/修改器方法。請記住，如果您發現簡單的資料屬性需要擴展功能行為，Python 提供了一條未來增強的簡單路徑。在這種情況下，請使用屬性將功能實作隱藏在簡單的資料屬性存取語法後面。

公共和內部介面

任何向後相容性保證僅適用於公共介面。因此，使用者能夠清楚地區分公共介面和內部介面非常重要。

記錄的接口被認為是公共的，除非文件明確聲明它們是臨時或內部接口，不受通常的向後相容性保證。所有未記錄的介面都應假定為內部介面。

為了更好地支援自省，模組應使用屬性在其公共 API 中明確聲明名稱__all__。設定 __all__為空列表表示該模組沒有公共 API。

即使__all__設定正確，內部介面（套件、模組、類別、函數、屬性或其他名稱）仍應以單一前導底線為前綴。

如果任何包含命名空間（套件、模組或類別）被視為內部的，則介面也被視為內部的。

導入的名稱應始終被視為實作細節。其他模組不得依賴對此類導入名稱的間接訪問，除非它們是包含模組的 API 的明確記錄部分，例如os.path或公開子__init__模組功能的包模組。

  • 不想使用類型檢查器的使用者可以忽略它們。但是，預計第三方庫包的用戶可能希望對這些包運行類型檢查器。為此， PEP 484建議使用存根檔案：類型檢查器優先於相應的 .py 檔案讀取的 .pyi 檔案。存根檔案可以與庫一起分發，也可以透過 typeshed repo [5]單獨分發（在庫作者的許可下） 。

變數註釋

PEP 526引入了變數註解。對它們的風格建議與上面描述的函數註釋的風格建議類似：

  • 模組級變數、類別和實例變數以及局部變數的註解應在冒號後面有一個空格。
  • 冒號之前不應有空格。
  • 如果賦​​值有右側，則等號兩側應恰好有一個空格：
# Wrong:

code:int  # No space after colon
code : int  # Space before colon

class Test:
    result: int=0  # No spaces around equality sign

儘管Python 3.6 接受PEP 526 ，但變數註釋語法是所有 Python 版本上存根檔案的首選語法（有關詳細信息，請參閱PEP 484）。
